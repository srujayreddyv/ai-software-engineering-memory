# Task Plan

## Task

State the requested change in one or two sentences.

## Objective

Define the desired outcome and how success will be recognized.

## Current State

Summarize the relevant existing behavior, files, architecture, patterns, and constraints.

## Proposed Solution

Describe the smallest change that should satisfy the objective while preserving existing architecture and conventions.

## Files Affected

List the files or directories expected to change and why each change is needed.

## Database Impact

State whether the task affects schema, migrations, data access, seed data, fixtures, or stored records. If there is no database impact, say so.

## Testing Strategy

List the tests to add, update, or run. Include the behavior each test verifies.

## Risks

Document correctness, security, performance, maintainability, compatibility, or operational risks.

## Rollback Plan

Explain how the change can be reverted or mitigated if it causes problems.

## Implementation Checklist

* Confirm assumptions.
* Make the planned code or documentation changes.
* Run the planned verification.
* Review the result against the objective.
* Update memory if durable repository knowledge changed.
